# Definition of Done

This is the shared agreement between you (the PO) and the dev team on what "Done" actually means. A story is not Done until every item on this list is true. This prevents the situation where a dev says "it's done" but you find problems when you go to check it.

---

## A story is Done when:

- [ ] Every acceptance criterion in the story has been met
- [ ] It works correctly on both mobile and desktop
- [ ] It has been tested — not just by the developer who built it
- [ ] No new bugs have been introduced by the change
- [ ] You (the PO) have reviewed it and signed it off
- [ ] It is deployed to a place where you can actually see and test it

---

## A story is NOT Done if:

- It works on desktop but not mobile
- It passes some acceptance criteria but not all
- Only the developer who built it has tested it
- It has been built but not yet deployed
- You have not reviewed it yet

---

## Who decides if something is Done?

**You do.** The dev team can say it is built — but only the PO signs it off as Done. If something does not meet the acceptance criteria, send it back. This is normal and expected.

---

## Why this matters

Without a clear Definition of Done, "done" means something different to everyone. The dev team thinks it's finished. You find problems. The sprint review becomes a disagreement. This document prevents that.

Think of it like a referee's call — the Definition of Done is the rulebook. Everyone plays by the same rules.
